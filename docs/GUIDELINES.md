# Harness Builder Team — 사용 가이드라인

> 빌더 팀을 올바르게 사용하기 위한 상세 지침입니다.

---

## 1. 기본 원칙

### 1-1. 트리거 키워드 시스템

빌더 에이전트는 **정확한 키워드**에만 반응합니다. 잘못된 키워드는 엉뚱한 에이전트를 트리거하거나 아무것도 트리거하지 않을 수 있습니다.

| 원하는 동작 | 올바른 입력 | 잘못된 예시 |
|------------|------------|------------|
| Phase 1 시작 | "하네스 세팅 진행해" | "시작해", "세팅" |
| Phase 3 시작 | "00-team-architect 진행해" | "팀 만들어줘" |
| MCP 설치 후 팀 강화 | **"MCP 재검증"** | "재검증" ← QA 트리거됨 |
| QA 검증 | "00-qa-gate 진행해" 또는 "재검증" | "확인해줘" |
| QA 재검증 | "재검증" 또는 "QA 재검증" | — |

> [!WARNING]
> **"재검증"** 단독 입력은 `00-qa-gate`를 트리거합니다.
> MCP 설치 후 팀을 강화하고 싶다면 반드시 **"MCP 재검증"** 을 입력하세요.

### 1-2. ONBOARDING.md 완료 후 동작

빌더 팀은 `ONBOARDING.md`가 존재하면 자동으로 비활성화됩니다.
- 하네스 세팅을 다시 하려면 `ONBOARDING.md` 삭제 후 재시작
- 타겟 에이전트만 재구성하려면 `01-*~` 폴더만 삭제 후 Phase 3부터 재시작

### 1-3. 온보딩 후 빌더 팀 사용 (Support Mode)

`ONBOARDING.md` 생성 후 빌더 팀은 **"눈치껏 대기하는 비서"** 모드로 전환됩니다.
- **자동 개입 금지**: 실무 개발 중에 빌더 팀이 먼저 말을 걸지 않습니다.
- **수동 호출 허용**: 특정 빌더 에이전트의 도움이 필요하면 이름을 직접 부르세요.
  - 예: *"00-qa-gate, 지금까지 만든 코드 다시 검사해줘"*
  - 예: *"00-doc-architect, PRD에 결제 기능 추가하고 문서 업데이트해줘"*

## 2. Phase별 상세 가이드

### Phase 1: 요구사항 수집 (00-harness-pm)

**언제**: 새 프로젝트를 시작할 때

**입력 방법**:
```
하네스 세팅 진행해
```

**흐름**:
1. 기존 `docs/PRD.md` 감지 → 있으면 분석, 없으면 `INTAKE_FORM.md` 생성
2. 기술 스택 모를 때: INTAKE_FORM에 "추천해줘" 기입 → PM이 최적 스택 제안
3. 브리핑 후 "예" 입력 시 Phase 2로 이동

---

### Phase 2: 문서 설계 (00-doc-architect)

**언제**: Phase 1 승인 후

**생성 문서**:
| 파일 | 역할 |
|------|------|
| `docs/PRD.md` | 제품 요구사항 명세 (변경 금지) |
| `docs/ARCHITECTURE.md` | 기술 아키텍처 정의 |
| `docs/ROADMAP.md` | 전체 Phase 전략 (완료 Phase는 1줄 요약으로 아카이빙) |
| `docs/PLANNING.md` | 매 턴 덮어쓰는 단기 실행 계획 |
| `docs/CHANGELOG.md` | 변경 이력 |

---

### Phase 3: 1차 팀 구성 (00-team-architect)

**언제**: Phase 2 완료 후

**출력**:
- 프로젝트에 맞는 타겟 에이전트 팀 (`.agents/skills/01-*~`)
- 설치 권장 MCP/Skills 추천 목록 (웹 검색 기반)

**Phase 3 완료 후 선택지**:
- **A) MCP 설치 후 팀 강화** → MCP 설치 → `"MCP 재검증"` 입력 (Phase 5)
- **B) 바로 진행** → `"00-security-enforcer 진행해"` 입력 (Phase 6 이동)

---

### Phase 4: MCP/Skills 설치 (사용자)

빌더 팀이 추천한 도구 중 원하는 것을 수동으로 설치합니다.
설치 후 반드시 **"MCP 재검증"** 을 입력하세요.

---

### Phase 5: MCP 재검증 + 팀 강화 (00-team-architect)

**언제**: MCP/Skills 설치 완료 후

**입력**: `"MCP 재검증"`

**동작**: 새로 감지된 도구를 타겟 에이전트 Instructions에 자동 주입

---

### Phase 6: 보안 강화 (00-security-enforcer)

**입력**: `"00-security-enforcer 진행해"`

**주입 내용**:
- API 키 하드코딩 금지
- `.env.local` 보안 프로토콜
- 파괴적 명령어 금지
- DB RLS 정책 강제 (DB 프로젝트)

---

### Phase 7: 최종 QA (00-qa-gate)

**입력**: `"00-qa-gate 진행해"`

**10개 검증 항목**:
1. 환각 방지 (Web Grounding) 포함?
2. 보안 네거티브 제약 포함?
3. SSOT 참조 지시 포함?
4. 에이전트 간 역할 분리 완료?
5. PRD ↔ 에이전트 역할 정합성?
6. MCP/Skills 활용 지시 반영?
7. Boundary (작업 범위) 명시?
8. description에 네거티브 트리거?
9. 보호구역(00-*) 접근 금지 제약?
10. **README 품질**: 프로젝트 기획 내용이 루트 리드미에 반영되었는가?

**재검증**: `"재검증"` 또는 `"QA 재검증"` — 언제든 수동 재호출 가능

---

## 3. 기획 변경 대응

PRD를 수정한 후 팀을 업데이트하려면:

1. `docs/PRD.md` 직접 수정
2. `"기획 변경 반영"` 입력 → `00-team-architect` Mode C 활성화
3. 영향받는 에이전트 수정 제안 → 승인
4. `"00-security-enforcer 진행해"` → `"재검증"` 순으로 재실행

---

## 4. 에러 복구

| 상황 | 해결 방법 |
|------|----------|
| Phase 중간에 실패 | 해당 Phase 에이전트를 다시 트리거 |
| 엉뚱한 에이전트 트리거됨 | 트리거 키워드 확인 후 재시도 |
| 타겟 에이전트가 잘못 생성됨 | `01-*` 폴더 삭제 후 Phase 3 재실행 |
| 보안 룰 누락 | Phase 6 재실행 |
| QA 반려 | 반려 사유 확인 → Phase 3 재실행 |

---

## 5. 커스터마이징

### 빌더 에이전트 수정

`00-*` 폴더의 `SKILL.md`를 직접 수정할 수 있습니다.
단, YAML frontmatter의 `name`과 `description`은 반드시 유지하세요.

### 타겟 에이전트 템플릿 수정

`00-team-architect/references/SKILL_TEMPLATE.md`를 수정하면
이후 생성되는 모든 타겟 에이전트에 반영됩니다.
